## Context

The Zalo Mini App runs inside a Zalo-owned `<iframe>`. When `statusBar: "transparent"` and `actionBarHidden: true` are set in `app-config.json`, the device OS status bar (iOS / Android) is drawn on top of the iframe without pushing it down. Similarly, the iOS home indicator and the Android system navigation bar can overlap the bottom of the content.

Zalo exposes two CSS custom properties inside the iframe:
- `--zaui-safe-area-top` — height of the top overlap (typically matches device status bar height)
- `--zaui-safe-area-bottom` — height of the bottom overlap (home indicator or nav bar)

Currently no layout component reads these variables.

## Goals / Non-Goals

**Goals:**
- Apply `--zaui-safe-area-top` to AuthLayout's LanguageSwitcher so it is not hidden under the status bar
- Apply `--zaui-safe-area-top` to AppLayout's page content so it is not obscured at the top
- Apply `--zaui-safe-area-bottom` to BottomNav so it clears the home indicator / Android nav
- Add reusable `.pt-safe` / `.pb-safe` CSS utility classes for future use

**Non-Goals:**
- Handling safe areas on non-Zalo platforms (the variables default to `0px` outside Zalo)
- Supporting `env(safe-area-inset-*)` (native iOS Safari — not relevant for Zalo iframe)
- Changing visual design, colors, or spacing beyond the safe-area offset

## Decisions

### Use CSS custom properties with `0px` fallback
`var(--zaui-safe-area-top, 0px)` degrades cleanly outside the Zalo iframe — layout is unaffected in browser dev and Playwright tests.

**Alternative considered**: Feature-detect and read the value in JS. Rejected — CSS is simpler, zero JS overhead, and eliminates flash of wrong position on mount.

### Utility classes in `app.css` rather than Tailwind config
The values are dynamic (vary per device at runtime), so they cannot be pre-generated by Tailwind's build step. Arbitrary value syntax like `pt-[var(--zaui-safe-area-top)]` works but is verbose and repetitive. Two short utility classes in `app.css` are cleaner and reusable.

**Alternative considered**: Tailwind plugin that adds the utilities. Overkill for two classes.

### Apply offset to BottomNav height, not AppLayout bottom padding
AppLayout's content area uses `overflow-y: auto` and the bottom nav is fixed-positioned. Adding bottom padding to the content area would double-compensate. The BottomNav itself shifts up via `bottom: var(--zaui-safe-area-bottom, 0px)`, and its own height includes the padding visually.

## Risks / Trade-offs

- **Variable not available in Playwright/Vitest tests** → Mitigation: default `0px` means layout renders normally; no test changes required.
- **Device-specific values change** → Mitigation: variables are set by the Zalo runtime, not hardcoded; fix automatically adjusts per device.

## Migration Plan

Pure CSS/layout change — no data migration, no API change, no feature flag. Deploy and it takes effect immediately. Rollback: revert the three component files and `app.css`.
